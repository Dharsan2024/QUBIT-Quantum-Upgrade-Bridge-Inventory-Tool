## 1. Purpose & Requirements

`qubit-bridge` is the subsystem that **proves** QUBIT's story end-to-end. Every other package produces analysis (an inventory, a risk score, a diff); the bridge produces the *observable physical outcome* an evaluation committee can watch on a packet capture: a service that was negotiating classical RSA-2048 + X25519 is, after remediation, negotiating **hybrid X25519MLKEM768**. It owns four things:

- **(a) Hybrid TLS termination** — reproducible nginx/HAProxy container images that terminate TLS 1.3 with the `X25519MLKEM768` hybrid key exchange in front of otherwise-classical backends.
- **(b) Verification tooling** — a client-side probe that connects to any host and reports the negotiated group / signature algorithm / certificate key type, plus pcap capture for demo proof, all mapped back to the shared `CryptoAsset` schema.
- **(c) The demo-lab** — deliberately-vulnerable sample apps (`vulnapp-python`, `vulnapp-java`; a Node/Go app is M3 stretch) + docker-compose scenario + a scripted **4-phase committee demo** (harvest → discovery/CBOM → risk dashboard → automated patch + hybrid re-capture). App names, flaws, and fixture tags are fixed by the shared **`demo-lab/SPEC.md`** (§6.5) co-owned with docs 01/03/05.
- **(d) Performance harness** — measures classical vs hybrid handshake latency and on-wire key-share sizes to produce the paper's evaluation tables.

### 1.1 Functional requirements

| ID | Requirement |
|----|-------------|
| FR-1 | `qubit bridge up <profile>` starts a hybrid TLS terminator (nginx or HAProxy) in front of a named upstream; the listener negotiates `X25519MLKEM768` with any OpenSSL-3.5 / Chrome / Firefox client. |
| FR-2 | `qubit bridge probe <host:port>` reports negotiated TLS version, key-exchange group (+ IANA codepoint), whether the group is a PQC hybrid, cipher suite, peer signature type, and certificate public-key algorithm/size. |
| FR-3 | Probe output is convertible to one or more `CryptoAsset` records (`source_scanner=network`, `asset_type=protocol`) and POSTable to the QUBIT API. |
| FR-4 | `qubit bridge verify <host:port> --expect X25519MLKEM768` exits 0 on match, non-zero on mismatch — usable as a CI/demo assertion. |
| FR-5 | `qubit bridge capture` records the TLS handshake to a `.pcap` and can diff a *before* (classical) vs *after* (hybrid) capture. |
| FR-6 | `qubit bridge bench` runs N handshakes per group and emits latency percentiles + key-share byte sizes as CSV/JSON. |
| FR-7 | `demo-lab` provides ≥3 vulnerable apps with intentional, scanner-detectable crypto flaws and classical TLS. |
| FR-8 | `qubit demo run --phase {1..4}` (or `--all`) executes the 4-phase demo from the **host CLI** and is idempotent (re-runnable); phase 4 uses `qubit migrate apply --auto-approve` for the scripted path. Not exposed as a REST endpoint (a containerized API driving docker would need a docker-socket mount). |
| FR-9 | Everything runs offline via `docker compose up` — no external network, no telemetry. |

### 1.2 Non-functional requirements

- **NFR-1 Reproducibility:** identical result on a clean checkout; all crypto material (certs/keys) generated by a committed, deterministic-enough script, never checked in as secrets.
- **NFR-2 Portability:** dev on Windows 11 (team's machines) / Linux CI; all crypto-version-sensitive work happens *inside* Linux containers so the host's OpenSSL is irrelevant.
- **NFR-3 No cloud crypto:** hybrid KEX uses OpenSSL 3.5 shipped in the container; no managed TLS, no cloud KMS.
- **NFR-4 Performance:** hybrid handshake overhead must be *measured*, not assumed — the harness is the deliverable, not a target number.
- **NFR-5 Licensing:** OpenSSL 3.x (Apache-2.0), nginx (BSD-2), HAProxy (GPL-2.0, used unmodified as a container binary — no linking, no import), liboqs/oqs-provider (MIT). Python deps are MIT/BSD; **scapy (GPL-2.0) is an optional extra `qubit-bridge[scapy]`, never a hard dependency** (pyshark/tshark-subprocess is the default). QUBIT code itself MIT.
- **NFR-6 Safety:** vulnerable apps bind only to the compose network / localhost, carry a `DELIBERATELY-VULNERABLE` banner, and never ship in the distributable Python wheels.

## 3. Exact Tech Stack (verified, 2026)

### 3.1 Hybrid KEX runtime — **native OpenSSL 3.5, oqs-provider optional**

**Decision:** use **OpenSSL 3.5.x** (3.5.0 GA 8 Apr 2025, LTS through Apr 2030) which ships ML-KEM/ML-DSA/SLH-DSA natively and enables the `X25519MLKEM768` hybrid group **as a default TLS 1.3 keyshare** — no third-party provider required. This is strictly simpler and more robust than the oqs-provider path for the *key exchange* and is what current browsers negotiate by default. `oqs-provider 0.11.0` + `liboqs 0.15.0` (stable; 0.16.0-rc1 in dev, Jun 2026) are retained only as an **optional** profile for (i) pure `MLKEM768` / (ii) exotic algorithms / (iii) ML-DSA-in-certificate experiments on hosts where native support is missing. See Frame deviation §11.

| Purpose | Component | Version pin (2026) | License |
|---------|-----------|--------------------|---------|
| TLS + PQC KEX engine | OpenSSL | ≥ 3.5.4 (via base image) | Apache-2.0 |
| Base image (nginx) | `nginx:<current>-trixie` (Debian 13, OpenSSL 3.5) **or** `nginx:alpine` (Alpine 3.22) | pinned by digest | — |
| Reverse proxy A | nginx | 1.26.3 (trixie distro) / newer via official trixie image | BSD-2-Clause |
| Reverse proxy B | HAProxy | 3.0.11 (trixie distro) / `haproxy:3.2-trixie` | GPL-2.0 (unmodified binary) |
| PQC provider (optional, exotic algs only) | oqs-provider | 0.11.0 | MIT |
| PQC library (optional, exotic algs only) | liboqs | 0.15.0 | MIT |

> **Important image note:** hybrid-group support comes from the linked **OpenSSL ≥ 3.5**, not the proxy version — so either the official **`nginx:*-trixie`** images (trixie-based, OpenSSL 3.5, available well before mid-2026, e.g. `nginx:1.29.3-trixie`) or the **Alpine 3.22** variant work; the old "nginx:1.29 is Debian-12/OpenSSL-3.0" claim is outdated. A build-time guard `openssl version | grep -Eq 'OpenSSL 3\.(5|6|7|8|9)'` fails the build if the base somehow ships < 3.5. (Distro nginx 1.26.3 / HAProxy 3.0.11 on trixie also work — the version numbers are not load-bearing.)

| Purpose | Library | Version | License | Notes |

**Probe transport — canonical choice:** shell out to **`openssl s_client`** *inside the bridge container* (guaranteed OpenSSL 3.5), parse stdout. **Networking (Windows-correct):** the probe container is attached to the demo **compose network** and targets the terminator by **service DNS** (`nginx-hybrid:8443`) — *not* `127.0.0.1`, which on Docker Desktop for Windows/macOS resolves to the probe container's own loopback (and `--network host` does not work there). For host-published ports the probe rewrites the target to `host.docker.internal`. This is codified in `probe.py` and covered by a Windows M1 acceptance test.

The stdlib `ssl` groups API (`ssl.SSLContext.set_groups()` / `SSLSocket.group()`) is the clean in-process probe, but it is a **Python 3.15** feature (verified: documented under Python 3.15 What's New; `SSLSocket.group()` additionally needs OpenSSL ≥ 3.2) — *not* 3.14. It is used only as an optional fast-path, gated on **`hasattr(ssl.SSLSocket, "group") and ssl.OPENSSL_VERSION_INFO >= (3,5)`** (version-proof regardless of interpreter), otherwise the `s_client` subprocess path runs. Note Python 3.14's own Windows/macOS installers bundle OpenSSL 3.5.x, so once 3.15 ships, students installing it get a clean native probe.

### 3.3 Demo-lab app stacks

| App | Language/runtime | Framework | Intentional flaws (each = a doc-03 rule-pack ID, tagged `# QUBIT-FIXTURE: <rule-id>`) |
|-----|------------------|-----------|--------------------------------------|
| `vulnapp-python` | Python 3.12 | Flask 3.x + PyJWT 2.x + cryptography | RSA-OAEP file encryption (`py-rsa-enc-01`); ECDH kex (`py-ecdh-kex-01`); ECDSA JWT signing (`py-ecdsa-sig-01`); SHA-1 password hash (`py-weakhash-01`); RSA-2048 TLS cert |
| `vulnapp-java` | Java 21 (Temurin) | Spring Boot 3.3 | JCA RSA keygen + `SHA256withECDSA` signing + `MessageDigest.getInstance("SHA-1")`; RSA-2048 keystore (Java migrates via **template path** at M2) |
| `vulnapp-node` (M3 stretch) | Node 22 | Express 4 + node-forge | node-forge RSA-2048; `crypto.createHash('sha1')` |

Flaws are chosen to match **rules that actually exist in the doc-03 pack** — note there is deliberately no RS256-JWT→PQC-JOSE flaw, because no mainstream JOSE library registers an ML-DSA `alg` in mid-2026, so there is nothing to automatically patch it to (JWT assets get an inventory recommendation only).

Each app serves plain HTTP on an internal port; **classical TLS** for phase 1 is provided by a **classical nginx profile** (symmetric with phase 4's hybrid nginx — one ingress pattern throughout, not the app's own server). The **hybrid** posture is applied by putting the *same* app behind the `nginx-hybrid` terminator (phase 4), the realistic ingress-controller migration pattern from the research plan — *required* for Java/Node whose native runtimes do not yet offer hybrid TLS (the JDK's PQC-hybrid TLS KEM JEP is targeted at a future JDK, not GA in 2026 — cite the JEP by number when confirmed).

## 4. Data Models / Schemas

All persisted crypto facts conform to the shared `CryptoAsset` (frame §"Shared CryptoAsset schema", `qubit-core`). Only **one** new persisted model moves to `qubit-core`: `HandshakeMeasurement` (§4.3), so the API/dashboard can read bench results. `ProbeResult` and `BridgeProfile` are transient, live in `qubit_bridge/models.py`, and are never persisted.

### 4.1 `ProbeResult` (transient, `qubit_bridge/models.py`)

```python
class ProbeResult(BaseModel):
    host: str
    port: int
    reachable: bool
    tls_version: str | None            # "TLSv1.3" | "TLSv1.2" | ...
    negotiated_group: str | None       # "X25519MLKEM768"
    group_codepoint: int | None        # 0x11EC == 4588
    hybrid_pqc: bool                    # True iff negotiated_group in HYBRID set
    cipher_suite: str | None           # "TLS_AES_256_GCM_SHA384"
    peer_signature_type: str | None    # "RSA-PSS" | "ECDSA" | "ML-DSA-65"
    cert_public_key_algorithm: str | None  # "RSA" | "EC" | "ML-DSA"
    cert_public_key_bits: int | None   # 2048
    cert_signature_algorithm: str | None   # "sha256WithRSAEncryption"
    cert_fingerprint_sha256: str | None
    cert_not_after: datetime | None
    offered_groups: list[str]          # groups the client was configured to offer
    error: str | None
    raw_output: str                    # full s_client transcript (evidence)
    probed_at: datetime
```

### 4.2 `ProbeResult → CryptoAsset` mapping (`qubit_bridge/assets.py`)

A single TLS probe yields **one protocol asset** and (optionally) **one certificate/key asset**:

```python
def probe_to_assets(r: ProbeResult) -> list[CryptoAsset]:
    proto = CryptoAsset(
        id=uuid4(),
        source_scanner="network",
        location={"host": r.host, "service": f"tcp/{r.port}"},
        asset_type="protocol",
        algorithm=r.negotiated_group or "unknown",          # canonical group name
        key_size=None,
        protocol_detail={
            "protocol": "tls",
            "version": r.tls_version,
            "cipher_suites": [r.cipher_suite] if r.cipher_suite else [],
            "group": r.negotiated_group,
            "group_codepoint": r.group_codepoint,
        },
        usage_context="kex",
        quantum_vulnerable={
            "vulnerable": not r.hybrid_pqc,                  # classical group => Shor-vulnerable
            "attack": "shor" if not r.hybrid_pqc else "none",
        },
        evidence=_pcap_or_transcript_ref(r),          # typed EvidenceRef (pcap path / transcript id), not a raw dict
        discovered_at=r.probed_at,
        risk=None, migration=None, sensitivity="unknown", shelf_life_years=None,
    )
    assets = [proto]
    if r.cert_public_key_algorithm:                          # server-auth cert as its own asset
        assets.append(CryptoAsset(
            id=uuid4(), source_scanner="cert",
            location={"host": r.host, "service": f"tcp/{r.port}"},
            asset_type="certificate",
            algorithm=_canon(r.cert_public_key_algorithm, r.cert_public_key_bits),  # "RSA-2048"
            key_size=r.cert_public_key_bits,
            usage_context="signature",
            quantum_vulnerable={"vulnerable": _is_classical_pk(r), "attack":
                                "shor" if _is_classical_pk(r) else "none"},
            evidence={"cert_fingerprint_sha256": r.cert_fingerprint_sha256},
            discovered_at=r.probed_at, risk=None, migration=None,
            sensitivity="unknown", shelf_life_years=None,
        ))
    return assets
```

Canonical algorithm names come from `qubit-core`'s algorithm registry (frame). The bridge's `registry.py` only owns the **group** table:

```python
# qubit_bridge/registry.py  (IANA TLS Supported Groups, verified)
HYBRID_GROUPS = {
    "X25519MLKEM768":     0x11EC,   # 4588  (default in OpenSSL 3.5 / Chrome / Firefox)
    "SecP256r1MLKEM768":  0x11EB,   # 4587  (FIPS-friendly)
    "SecP384r1MLKEM1024": 0x11ED,   # 4589
}
PURE_PQC_GROUPS = {                 # natively supported by OpenSSL 3.5 (NOT "only via oqs-provider")
    "MLKEM512": 0x0200, "MLKEM768": 0x0201, "MLKEM1024": 0x0202,
}
CLASSICAL_GROUPS = {
    "x25519": 0x001D, "secp256r1": 0x0017, "secp384r1": 0x0018, "x448": 0x001E,
}
def is_hybrid(group: str) -> bool: return group in HYBRID_GROUPS
def codepoint(group: str) -> int | None: return {**HYBRID_GROUPS, **PURE_PQC_GROUPS, **CLASSICAL_GROUPS}.get(group)
```

## 6. Key Algorithms & Flows

### 6.1 Probe (client-side handshake fact extraction)

```
probe_host(host, port, groups):
    cmd = ["openssl","s_client","-connect", f"{host}:{port}",
           "-tls1_3", "-brief", "-servername", sni or host]
    if groups: cmd += ["-groups", groups]          # force-offer only these groups
    proc = run(cmd, input=b"", timeout=timeout)    # empty stdin => close after handshake
    txt  = proc.stdout + proc.stderr
    if connect failed: return ProbeResult(reachable=False, error=..., raw=txt)
    result.tls_version       = grep r"Protocol version: (TLS.*)"  OR "TLSv1.3" from -brief
    result.negotiated_group  = grep r"Negotiated TLS1.3 group:\s*(\S+)"        # primary
                               || grep r"Server Temp Key:\s*(\S+)"             # fallback
    result.cipher_suite      = grep r"Ciphersuite:\s*(\S+)"  || r"Cipher\s*:\s*(\S+)"
    result.peer_signature_type = grep r"Peer signature type:\s*(.+)"
    # certificate facts: re-run with -showcerts OR parse `openssl x509` on the leaf
    leaf = extract_first_cert(txt)
    if leaf:
        x = run(["openssl","x509","-noout","-text","-fingerprint","-sha256"], input=leaf)
        result.cert_public_key_algorithm, result.cert_public_key_bits = parse_pubkey(x)
        result.cert_signature_algorithm = parse_sigalg(x)
        result.cert_fingerprint_sha256  = parse_fpr(x)
        result.cert_not_after           = parse_notafter(x)
    result.group_codepoint = registry.codepoint(result.negotiated_group)
    result.hybrid_pqc      = registry.is_hybrid(result.negotiated_group)
    return result
```

Parser is pure and fixture-tested (§8). The parser reads the `Negotiated TLS1.3 group:` line (present in OpenSSL 3.5 `s_client` **default** output) with `Server Temp Key:` as a fallback; whether `-brief` also emits that exact line is confirmed by the M1 golden fixture before we rely on it (don't assume `-brief` format until the fixture proves it).

### 6.2 Bring-up of a hybrid terminator

```
bring_up(profile):
    cfg = render_config(profile)                 # Jinja2 -> nginx/haproxy text
    write cfg to ./generated/{engine}.conf
    ensure certs exist (call gen-certs.sh if missing)
    docker compose -f compose.hybrid.yml up -d {engine}-hybrid
    wait_until(port open, <=20s)
    r = probe_host(f"{engine}-hybrid", profile.listen_port, groups="X25519MLKEM768")  # service DNS, not 127.0.0.1
    assert r.negotiated_group == "X25519MLKEM768", "bridge failed to negotiate hybrid"
```

### 6.3 Handshake benchmark

```
bench_group(host, port, group, n):
    samples_ms = []
    for i in 1..n:
        t0 = perf_counter()
        run(["openssl","s_client","-connect",f"{host}:{port}","-tls1_3",
             "-groups",group,"-brief"], input=b"")
        samples_ms.append((perf_counter()-t0)*1000)
    m = HandshakeMeasurement(
        group=group, hybrid_pqc=is_hybrid(group), n_samples=n,
        handshake_ms_mean=mean(samples_ms), p50=median, p95=quantile(.95),
        stdev=stdev(samples_ms), ...)
    # sizes: one capture handshake, parse ClientHello/ServerHello key_share ext length
    pcap = capture_handshake(host, port, tmp, handshakes=1, group=group)
    m |= extract_key_share_sizes(pcap)           # scapy: TLS_Ext_KeyShare -> len(key_exchange)
    return m
```

`extract_key_share_sizes` walks the pcap for the `key_share` extension in ClientHello/ServerHello and records the `key_exchange` byte length. **Byte sizes are the harness's PRIMARY metric** — deterministic and RTT-independent: `X25519MLKEM768` client share ≈ **1216 B** vs `x25519` ≈ **32 B**, a ~1.1 KB flight increase, the headline evaluation number.

> **Latency methodology (valid, not process-spawn noise):** timing the whole `s_client` subprocess measures process spawn (tens of ms, high variance) against a sub-millisecond hybrid-vs-classical delta — noise ≫ signal — and localhost RTT≈0 hides the extra-bytes effect entirely. So handshake duration is derived from **pcap timestamps** (first ClientHello → client Finished), and the compose scenario injects realistic RTT with **`tc netem`** (10/50/100 ms profiles) so the extra flight actually costs something measurable. An in-container Python 3.15 `PySSLRunner` is the in-process timing source when available. This is a **never-cut M3 deliverable** — the paper's overhead section depends on it (it moved *up* the never-cut list from the old cut-line #6).

### 6.4 4-phase committee demo (`qubit demo run --all`)

Every phase supports **`--canned`** (see §6.4.1) so the bridge track's own rehearsals and M2 acceptance pass even when a sibling subsystem is not yet live.

```
Phase 1 — HARVEST (classical baseline)
  docker compose -f compose.classical.yml up -d        # vulnapp-python behind CLASSICAL nginx on :8443
  qubit bridge capture nginx-classical:8443 --out out/harvest_classical.pcap --handshakes 1
  qubit bridge probe   nginx-classical:8443             # shows group=x25519, cert=RSA-2048
  => narration: "adversary stores this ciphertext; key exchange is Shor-breakable"

Phase 2 — DISCOVERY / CBOM
  qubit scan demo-lab/vulnapp-python --cbom out/cbom.json   # qubit-scanner (reads GET /scans/{id}/cbom)
  qubit bridge probe nginx-classical:8443 --push        # network asset -> POST /assets/batch
  => show terminal: file:line crypto findings + CycloneDX v1.7 CBOM JSON

Phase 3 — RISK
  (dashboard already reading DB) open http://localhost:3000/risk
  => Bayesian CRQC timeline curve + XGBoost priority queue; TLS-x25519 + ECDSA-JWT ranked

Phase 4 — REMEDIATE + HYBRID RE-CAPTURE (same port, no collision)
  qubit migrate apply --patch <py-ecdh-kex-01> --auto-approve    # qubit-migrate: LLM diff on a supported rule
  docker compose -f compose.classical.yml stop nginx-classical  # FREE :8443 first
  qubit bridge up hybrid --engine nginx --upstream vulnapp-python:5000 --port 8443  # SAME port
  qubit bridge capture nginx-hybrid:8443 --out out/harvest_hybrid.pcap --handshakes 1
  qubit bridge verify  nginx-hybrid:8443 --expect X25519MLKEM768   # PASS, exit 0
  qubit bridge probe   nginx-hybrid:8443 --push                    # re-scan: asset now non-vulnerable
  qubit scan demo-lab/vulnapp-python --no-db --json | grep -q ML-KEM   # code re-scan proves PQC present
  # migration.status -> verified for the affected assets
  qubit bridge diff out/harvest_classical.pcap out/harvest_hybrid.pcap
  => narration: "same service, same port, now negotiating X25519MLKEM768 — HNDL mitigated"
```

Phase 4 explicitly **stops the classical terminator before binding the hybrid one on the same port** — never two listeners racing for 8443 (that silent port-hop is exactly what F-8 now forbids in demo mode). It also **re-scans and `--push`es** so the dashboard/CBOM flip the asset to `verified` — the frame's "verified by re-scan and packet capture" requirement, not just a green probe.

A single committed file is the source of truth for the demo lab, so no two subsystems disagree about app names or flaws. It fixes:

| Field | Value |
|---|---|
| Apps (M2) | `vulnapp-python` (Flask), `vulnapp-java` (Spring Boot); `vulnapp-node` is M3 stretch |

## 7. Failure Modes & Handling

| # | Failure | Detection | Handling |
|---|---------|-----------|----------|
| F-1 | Base image has OpenSSL < 3.5 | build-time `openssl version` guard | image build fails loudly with remediation hint (use `debian:trixie`/`nginx:alpine 3.22`) |
| F-2 | Server can't negotiate hybrid; falls back to `x25519` | probe `negotiated_group != expect` | `verify` returns FAIL + non-zero exit; message names actual group |
| F-3 | Target not TLS 1.3 (hybrid is 1.3-only) | `tls_version != TLSv1.3` | probe reports version; `hybrid_pqc=False`; asset flagged `quantum_vulnerable` |
| F-4 | Host unreachable / timeout | subprocess timeout / connect error | `ProbeResult(reachable=False, error=...)`; CLI exit 2; no asset written |
| F-5 | `openssl s_client` binary/flag missing on host | `FileNotFoundError` / unknown-option | probe runs in a container attached to the compose network targeting service DNS; if Docker absent, actionable error |
| F-13 | Offline demo day — images not present on the machine | pre-flight image check | `make demo-bundle` (`docker save` all demo images → tarball) shipped ahead of time; `docker load` restores them; no registry pull needed at demo time |
| F-6 | pcap capture lacks privilege (`CAP_NET_RAW`) | tcpdump permission error | capture sidecar container runs with `cap_add: [NET_RAW]`; on host, document `sudo`/npcap; degrade to latency-only bench |
| F-7 | tshark/scapy TLS dissection unavailable | import/parse error | size fields set `None`, latency still reported; warn once |
| F-8 | Port 8443 already bound | compose up error | **In demo mode: hard error** ("stop the classical terminator first") — never a silent port-hop, which would make `verify :8443` hit the wrong listener in front of the committee. In non-demo mode only, `compose.py` may pick the next free port and echo it |
| F-9 | Self-signed cert expired/absent | `x509` parse / TLS alert | `gen-certs.sh` regenerates with 825-day validity; probe still records handshake facts |
| F-10 | QUBIT API unreachable on `--push` | httpx connect error | write assets to local `out/assets.json`, exit 0 with warning (don't fail the demo) |
| F-11 | oqs-provider ABI mismatch with OpenSSL (optional profile) | provider load error | native profile is default; optional profile marked experimental, isolated to its own image tag |
| F-12 | Java/Node app can't do hybrid TLS itself | by design | always front with `nginx-hybrid`; documented as the ingress-migration pattern, not a bug |

## 8. Testing Strategy

**Coverage target:** ≥70% on `qubit-bridge` (frame CI gate). Split into pure-unit (fast, no Docker) and integration (Docker, gated).

### 8.1 Unit tests (no network, no Docker)
- **Parser tests** (`test_probe_parse.py`): feed committed golden `s_client` transcripts (hybrid, classical, TLS1.2-fallback, RSA cert, EC cert) through the parser; assert every `ProbeResult` field. This is the highest-value test surface — the parser is the brittle part.
- **Mapping tests** (`test_assets.py`): `probe_to_assets` produces schema-valid `CryptoAsset`s; classical group ⇒ `quantum_vulnerable.attack == "shor"`; hybrid ⇒ `"none"`.
- **Registry tests**: codepoint/hybrid lookups, incl. `X25519MLKEM768 == 0x11EC`.
- **Config render tests** (`test_compose.py`): rendered nginx/haproxy text contains the `X25519MLKEM768` group directive for a given `BridgeProfile` (golden-file compare).
- **Size extraction tests**: run `extract_key_share_sizes` against 2 committed tiny `.pcap` fixtures (one classical, one hybrid); assert sizes within expected bands.

### 8.2 Integration tests (marked `@pytest.mark.docker`)
- **Live handshake** (`testcontainers`): build/start `nginx-hybrid`, `probe_host` it, assert `negotiated_group == "X25519MLKEM768"` and cert RSA-2048.
- **Verify exit codes**: `verify` PASS→0, mismatch→non-zero (start a classical profile to force mismatch).
- **Round-trip**: `bring_up` → `probe` → `probe_to_assets` → `POST /api/v1/assets/batch` on a spun-up `qubit-api` (SQLite) → GET assets back.
- **Demo smoke** (nightly, not per-PR): `qubit demo run --all --pcap-dir tmp`; assert both pcaps exist and `diff` shows a group change.

### 8.3 How fixtures are built
- **Certs:** `demo-lab/scripts/gen-certs.sh` — `openssl req -x509 -newkey rsa:2048 -sha256 -days 825 -nodes -subj "/CN=demo.local" -addext "subjectAltName=DNS:demo.local"`. Deterministic filenames, gitignored keys; CI regenerates on the fly.
- **Golden transcripts:** a `make fixtures` target runs `openssl s_client` against the freshly built `nginx-hybrid` and a classical nginx, capturing stdout to `tests/fixtures/*.txt`. Committed once, refreshed intentionally (never auto-regenerated in CI, so drift is a reviewable diff).
- **pcap fixtures:** `tcpdump -c 20 -w tests/fixtures/hybrid_handshake.pcap` during that same run; trimmed to the handshake with `editcap`/`tshark`. Small (<50 KB) so they belong in git.
- **CI runner note:** GitHub Actions `ubuntu-latest` ships OpenSSL 3.0, so all crypto-dependent integration tests run **inside the `nginx-hybrid` container** (via testcontainers) — the host OpenSSL is never exercised for hybrid KEX.

## 10. Risks, Mitigations & Cut-Lines

### 10.1 Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| OpenSSL-3.5 base image not reliably available for the students' setup | Med | High | Pin two known-good bases (Debian trixie **and** Alpine 3.22) by digest; build in CI early (M1) so drift surfaces immediately |

| Windows dev machines can't run hybrid client natively | High | Low | All crypto runs in containers; probe falls back to `docker run … openssl` |
| Java/Node can't do hybrid TLS in-runtime (JEP 527 = JDK 27, not GA) | High | Low | By design: front every app with `nginx-hybrid` ingress terminator — this *is* the migration story |
| pcap capture privilege issues on host | Med | Low | Dedicated capture sidecar with `cap_add: NET_RAW`; degrade to latency-only |
| oqs-provider / liboqs version churn breaks optional profile | Med | Low | Optional profile isolated to its own image tag; native OpenSSL path is the product default |
| Demo fails live in front of committee | Low | High | `qubit demo run --all` is idempotent, rehearsed in M2, and produces artifacts on disk; keep a recorded fallback pcap+screencast |

### 10.2 Explicit cut-lines (drop in this order under time pressure)

The **product story survives** as long as one app can be shown moving from classical → `X25519MLKEM768` on a packet capture. Drop from the bottom up:

1. **`vulnapp-node`** app — never in the core 2; delete first.
2. **oqs-provider profile / ML-DSA-in-cert experiment** — native OpenSSL hybrid KEX carries the paper; exotic-algorithm experiments are stretch.
3. **HAProxy engine** — keep **nginx only**; one working terminator is enough.
4. **`vulnapp-java`** — collapse demo-lab to **`vulnapp-python`** only (Python is the students' strongest stack and integrates most cleanly with `qubit-scanner`).
5. **In-process `PySSLRunner` fast-path** — keep `s_client` subprocess only.
6. **`bench_matrix` breadth** — benchmark just `{x25519, X25519MLKEM768}` (the one comparison the paper actually needs).

**Never cut:** the `nginx-hybrid` image, `probe`, `verify`, `vulnapp-python`, the same-port phase-1/phase-4 before/after capture, **and the pcap-timestamp + byte-size measurement** — the byte sizes are the paper's headline overhead number and are deterministic, so they stay even when latency timing is thin.

## 11. Frame Deviations

1. **KEX runtime: native OpenSSL 3.5 as default, not oqs-provider.** The frame lists "liboqs + oqs-provider (OpenSSL 3.x)". OpenSSL **3.5 (LTS, Apr 2025)** ships ML-KEM/ML-DSA/SLH-DSA natively and negotiates `X25519MLKEM768` by default; oqs-provider 0.11 in fact **auto-disables** anything OpenSSL ≥3.5 implements natively. We therefore use **native OpenSSL 3.5** for hybrid KEX, pure-ML-KEM groups, and ML-DSA certificates, and keep oqs-provider only for genuinely non-native algorithms (HQC/FrodoKEM-in-TLS). Satisfies the frame's intent with lower risk.
2. **Probe transport.** The frame implies Python-native tooling; because host Python's `ssl` is not linked to OpenSSL 3.5 on the team's Windows machines, the canonical probe shells out to `openssl s_client` in a container on the compose network (service DNS), with the stdlib `ssl` groups API (a **Python 3.15** feature + OpenSSL ≥3.2) as an optional fast-path gated on `hasattr(ssl.SSLSocket, "group")`. Behavior and outputs are identical; only the transport differs.
3. **Additive schema surface.** `qubit-bridge` adds one qubit-core table (`HandshakeMeasurement`, bench results) and two additive `protocol_detail` fields (`group`, `group_codepoint`). Additive only — no binding `CryptoAsset` field changes semantics.

All three deviations are additive and keep `qubit-bridge` fully within the frame's module-boundary and CryptoAsset-schema rules.
